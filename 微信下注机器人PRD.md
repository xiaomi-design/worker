## 2. 项目背景与目标

### 2.1 背景

当前微信群下注场景中，下注数据的采集、解析、结算及报表生成依赖人工操作，效率低且容易出错。需要一套自动化系统来替代人工流程。

### 2.2 目标

- 实现微信群下注消息的自动监听与采集
- 自动识别并解析多种下注格式
- 实现自动开奖与结算
- 自动生成每局 / 每日报表
- 支持 Windows 和 macOS 本地运行

| 角色 | 描述 |
|------|------|

| 玩家 | 在微信群中发送下注消息 |

## 4. 功能需求

### 4.1 微信群消息监听

- **描述**：系统监听指定微信群的聊天消息，实时获取玩家下注内容
- **优先级**：P0
- **验收标准**：系统能稳定监听微信群消息，延迟 ≤ 2 秒

### 4.2 下注内容解析

- **描述**：自动识别下注消息，支持以下格式：
  - 单点下注（1-6）
  - 保规则（`/`），最多保两个子
  - 载规则（`//`），最多载一个子
  - 多下注（`///`）
  - 三码下注（如 `123`）
- **优先级**：P0
- **验收标准**：所有合法格式均能正确解析，非法格式给出提示

### 4.3 下注数据记录

- **描述**：解析成功后自动记录下注数据，包括用户、下注内容、金额等
- **优先级**：P0
- **验收标准**：数据完整无遗漏，支持查询历史记录

### 4.4 开奖功能

- **描述**：支持 1-6 开奖，管理员手动触发或系统自动触发
- **优先级**：P0
- **验收标准**：开奖结果随机公正，开奖后自动触发结算

### 4.5 自动结算

- **描述**：根据下注规则和开奖结果自动计算输赢，规则如下：

| 下注类型 | 格式示例 | 赔率规则 |
|----------|----------|----------|
| 单点下注 | `1` | 点 1 中奖赔率 4 倍；点 2-6 中奖赔率 5 倍 |
| 保规则 | `1/2`（最多保两个子） | 中 1：3.2 倍；中 2：退本金；其他：输 |
| 载规则 | `1//2`（最多载一个子） | 中 1：2.4 倍；中 2：2.0 倍；其他：输 |
| 多下注 | `1///2///3` | 等价于多个独立下注 |
| 三码下注 | `123` | 包含 1 中奖：1.7 倍；不含 1 中奖：2 倍 |

- **优先级**：P0
- **验收标准**：结算金额计算准确，结果自动推送至微信群

### 4.6 报表生成

- **描述**：支持生成每局和每日汇总报表，格式为 CSV 或 Excel
- **优先级**：P1
- **报表字段**：用户、下注内容、金额、开奖结果、输赢、盈亏
- **验收标准**：报表数据与实际结算一致，支持导出下载

---

## 5. 系统流程

```
开始新局 → 玩家下注（微信群） → 系统解析下注 → 封盘 → 开奖 → 自动结算 → 输出报表
```

---

## 6. 非功能需求

| 需求项 | 要求 |
|--------|------|
| 平台兼容性 | 支持 Windows 和 macOS |
| 数据存储 | 使用 SQLite 或 CSV 进行本地存储 |
| 稳定性 | 系统连续运行不崩溃，异常自动恢复 |
| 消息延迟 | 从消息发出到系统解析完成 ≤ 2 秒 |

---

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 微信风控风险 | 账号被封禁，系统无法运行 | 控制消息频率，使用备用账号，做好账号隔离 |
| 群消息格式不规范 | 解析失败，下注遗漏 | 增加容错解析能力，对无法识别的消息给出提示 |
| 赌博相关法律风险 | 法律合规问题 | 仅限内部娱乐使用，做好合规审查 |
